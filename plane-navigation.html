Traversal, excavation, and saved return paths

You don’t “scroll” Atlas. You dig, orbit, and re-enter.

Navigation in Atlas is built around two ideas: (1) the top plane where you see the world’s structure, and (2) the depth carousel where you excavate meaning layer by layer. Every selected image can be saved into a named collection with the exact arrangement that produced it, so you can reproduce the same location even after catastrophic data loss.

I. Excavation

Three dig modes: needle, core sample, geological survey

Excavation decides what the carousel is “allowed” to consider when you dig. Same plane, different instrument.

Mode

Single (needle)

Only the selected column counts. The carousel returns a pure vertical stack. Highest precision, clean provenance.

Mode

Radius (core sample)

Include neighbors in rings. Ring 0 is the tower. Ring 1 is the first halo around it. Ring 2 is the halo of the halo. This reveals bridges and blended regions.

Mode

Layer-wise (survey)

Instead of digging one tower, you slice the whole world at a given depth index. “Show me the plane at depth 12.”

What changes
The plane doesn’t change. The meaning of the depth carousel changes: Single reads one tower, Radius reads a neighborhood, Layer-wise reads the entire world layer.
Why it matters
This is how you avoid two bad extremes: a sterile “one result” system and a chaotic “everything at once” system. You choose the aperture.

Atlas rule: the user controls the aperture of meaning. The same coordinate can be read like a scalpel, a shovel, or a satellite image.

II. Always-visible context

The Selection Stamp (1,2,3…) is the locator

Every selection displays a compact “stamp” that never disappears. It’s the human-readable breadcrumb that makes saves re-enterable.

Stamp = 1
Selected image only. No neighborhood elevated. Purest needle view.
Stamp = 2
Selected + ring-1 neighborhood. First halo included in excavation context.
Stamp = 3
Selected + ring-2 neighborhood. Two halos included (broader blend).
Selection Stamp = neighborhood rings included
1 → ring 0 only
2 → ring 0 + ring 1
3 → ring 0 + ring 1 + ring 2
… and so on
Why it’s non-negotiable

It makes recovery realistic (not mystical)

A saved coordinate alone can drift if the world rearranges. But a saved coordinate + excavation mode + stamp + depth index re-finds the same “kind of place.”

Guarantee

Re-enter the same context

Even if exact placement changes, you can return to the closest matching neighborhood and stratum.

Result

Saved items are maps

You didn’t save a picture. You saved a route: a reproducible path through the system.

Feel

“I’ve been here before.”

Because you have. The stamp makes déjà vu mechanical, not performative.

III. Saving

Save stores the world-state that produced the find

Each excavated image can be saved into a named collection. The save includes enough information to reconstruct the arrangement, then return to the same location (or the nearest match) later.

Saved with each item

Collection
User-provided name (folderless, but intentional).
Selected coordinate
(x,y,z) of the specific image, plus its depth index.
Excavation context
Mode (single/radius/layer-wise) + Selection Stamp (1,2,3…).
Note
Optional text annotation attached to the save.

The arrangement snapshot

Founding seeds
Which images/tags/clusters founded the plane or region.
Points on the plane
All placed points and their (x,y) positions.
Heat
Heat of each point (and any combined heat logic, if applicable).
Composition
What each point is made of: image, tag, tag-cluster, or combinations.
Dataset
The database/archive identifier used to populate the world.
SaveItem =
- id, collectionName, note
- selectedImageId, selectedXYZ, depthIndex
- excavationMode, radiusRings, selectionStamp
- snapshot: { seeds, points[], heat[], compositions[], datasetId }
→ enough to rebuild + re-enter
IV. Recovery

How you return after drift or total loss

Recovery is a replay: re-load the dataset, re-apply the snapshot, then navigate back using the saved excavation context. If the world isn’t identical, you land at the closest stable match.

Step 1
Load the dataset referenced by the save.
Step 2
Apply the snapshot: seeds, point positions, heat values, compositions.
Step 3
Locate the saved (x,y,z) coordinate and set depth to the saved depth index.
Step 4
Re-apply excavation mode + selection stamp to restore the original aperture of meaning.
Outcome
Same place if possible, closest match if not. In both cases: same *context*.

Core idea: the saved item is not just a bookmark. It’s a replayable instrument configuration. Atlas re-enters places the same way a musician replays a patch.
